---
title: How it works
description: From a plain-English question to a verified answer, in five stages.
---

Every question runs through the same pipeline. Each stage exists to catch a
specific failure mode, and the whole thing is wrapped in a self-correction loop.

<Steps>
<Step>
### Retrieval

DataPilot embeds your question and compares it against an embedding of every
table (built from its name, columns, sample values, and a generated
description). Only the handful of tables relevant to the question move forward -
the model never sees your entire schema, which keeps prompts small and answers
focused.
</Step>
<Step>
### Generation

The relevant tables plus your question become a prompt, and the model returns a
candidate SQL query. On a retry, the previous query and the exact error are
included too.
</Step>
<Step>
### Validation

The SQL is parsed to an abstract syntax tree. It's accepted only if the root is
a single `SELECT` **and** every table and column it references actually exists.
Anything else is rejected here - before it can run.
</Step>
<Step>
### Execution

The validated query runs on a read-only session with a statement timeout and a
row cap. Success returns rows and column types; a database error is captured as
feedback.
</Step>
<Step>
### Presentation

A chart type is chosen deterministically from the result's shape, a short
natural-language explanation is written, and the executed SQL is returned
alongside - never hidden.
</Step>
</Steps>

## The self-correction loop

The stages above form a loop, not a straight line.

```text
                 ┌──────────────── retry with the exact error ───────────────┐
                 ▼                                                            │
  question → retrieval → generation → validation → execution → presentation  │
                                          │            │                      │
                                          └────────────┴── failed? ───────────┘
                                                     (up to 3 attempts)
```

When validation or execution fails, the failure - **including the real column
names** when the model hallucinated one - is fed back into the next generation
attempt. Naming the available columns is what makes retries actually converge
instead of guessing again.

<Callout type="info">
  A `SELECT` that returns **zero rows is a success**, not a failure - it's a
  real answer, and the loop never retries it.
</Callout>

The response includes an `attempts` array, so a recovery is visible: more than
one attempt means the engine fixed its own mistake.

## Why retrieval, not "send the whole schema"

A database can have hundreds of tables. Sending all of them to the model on
every question is slow, expensive, and dilutes the context. DataPilot indexes
the schema once at connect time and, per question, retrieves only what's
relevant - plus any tables joined to those by a foreign key, so joins still
work.
